# A 股短线量化建议系统项目规约

## 1. 项目定位

本项目是一个面向个人使用的沪深 A 股和场内 ETF 短线量化决策辅助系统。系统目标是在不自动下单、不控制真实交易客户端的前提下，读取手动持仓台账、本地观察池和行情数据，生成可解释的买入、卖出、加仓、减仓、观望建议，并通过本地输出和后续消息渠道提醒用户。

系统只做个人研究和决策辅助，不承诺正收益。所有真实交易必须由用户在交易软件中人工确认。

## 2. 首版目标

首版以“能稳定生成可追溯建议”为核心，不追求复杂模型和自动化交易。

- 维护并读取手动持仓台账，将其作为真实持仓的唯一权威源。
- 维护并读取手动资金账户，将其作为现金余额、净本金和可用买入资金的唯一权威源。
- 只读获取东方财富自选候选，或按名称/代码搜索沪深 A 股和场内 ETF；用户确认后才增量加入本地观察池。
- 使用 AkShare 等公开数据源获取行情、历史 K 线、技术指标、资金流和分时强弱数据。
- 组合手动持仓台账、手动资金账户和行情快照生成账户估值。
- 收盘后生成次日交易计划。
- 盘中仅对持仓和计划内股票做触发判断。
- 通过统一 `DecisionWorkflow` 固化同一轮行情输入，并串联计划、策略、风控、建议、通知和审计。
- 输出本地控制台消息、Web/API 通知、账户快照和 JSONL 结构化日志；系统故障和永久失败邮件也必须进入同一套本地告警闭环。
- 使用本地 SMTP outbox 提供即时动作邮件、收盘摘要和系统告警；邮件失败不得影响决策工作流。
- 提供本地 Web 行情决策工作台，用于查看概览、K 线、资金流、分时强弱和数据引用。
- Telegram、飞书、钉钉等其他远程渠道继续作为后续范围。
- 东方财富模拟组合可在后续作为自动模拟操作决策层，用于演练策略建议，不作为真实持仓来源。

## 3. 非目标

以下能力不属于首版范围：

- 自动下单、模拟点击交易客户端、绕过人工确认。
- 直接读取真实交易账户的敏感信息。
- 依赖东方财富模拟组合推断真实持仓成本、数量或可用数量。
- 使用无法解释的黑盒模型作为主要决策依据。
- 全市场高频扫描。
- 原生移动 App、专业交易终端、画线工具和用户自定义指标。
- 在前端计算技术特征、策略、风控或建议。
- 承诺收益率、胜率或资金安全。

## 4. 运行形态

项目必须同时支持后台常驻服务、CLI 和认证 HTTP API，不能为不同入口维护互相矛盾的业务逻辑。后台调度器、CLI 和 HTTP 手动触发必须共同调用 `DecisionWorkflow`；该服务只负责编排，数据映射、特征、策略、风控、建议和通知继续由各自领域 service 负责。

后台服务负责：

- 使用 `XSHG` 交易日历按交易时段运行。
- 盘中每 3 分钟统一完成报价/分钟采集、账户估值、强弱、策略、风控、建议和通知。
- 在交易日 `15:15-16:30 Asia/Shanghai` 检查收盘数据就绪并幂等发布次日计划。
- 每个交易日 `16:35` 清理超出保留期的原始分钟线。
- 每 15 秒轮询邮件 outbox，且不受交易时段限制。
- 在启动时恢复已持久化的调度状态；若服务于交易日 `15:15-16:30` 重启且当日计划尚未发布，无论可选的普通 `run_on_start` 是否启用，都立即执行收盘就绪检查。
- 写入审计日志。
- 发送本地和远程推送。

CLI 负责：

- 检查配置和数据源状态。
- 手动新增、更新、删除、导入和查看持仓台账。
- 初始化、查看和维护手动资金账户，记录模拟银证转入、模拟银证转出和现金校准。
- 手动预览东方财富自选候选、搜索证券并确认加入观察池；预览不修改本地观察池或东方财富账户。
- 手动拉取行情数据。
- 手动生成账户估值快照。
- 通过统一收盘工作流生成次一交易日计划，并支持带原因和日历约束的人工补跑。
- 查看当前建议。
- 回放某日数据用于复盘。
- 通过 `qt workflow intraday` 手动触发盘中工作流；旧 `qt recommendations scan` 已退役并以非零状态退出。
- 非交易时段可通过 `qt workflow intraday --display-only` 显式刷新展示行情；该参数不等同于 `--force`，不会生成账户快照、计划、建议、通知或邮件。
- 回填日 K 和资金流基线、查看工作流与数据质量状态。
- 显式补跑收盘或盘中工作流、清理超期原始分钟线。
- 查看和标记通知、查看 SMTP 脱敏状态、测试邮件并重试失败投递。

### HTTP API

后台服务提供 `/api/v1` 认证 HTTP API，覆盖台账、资金、证券目录、候选预览与人工选择、股票池、行情与数据引用、计划、建议、通知、审计、调度、SMTP 和邮件 outbox。API 只返回后端已经计算并保存的证券元数据、特征、质量、理由和引用，不允许 Web 复制策略或风控。计划、建议、通知、反馈、审计和邮件投递等列表统一使用 `page/page_size`，并返回 `{items,total,page,page_size}`；旧 `POST /api/v1/watchlist/pinned/sync` 固定返回 HTTP `410 watchlist_sync_payload_retired`，旧 `POST /api/v1/recommendations/scan` 固定返回 HTTP `410 recommendation_scan_retired`，唯一盘中建议写入口是认证的 `DecisionWorkflow`。

服务按用户要求默认监听 `0.0.0.0` 以支持局域网访问，但不得未经 TLS、反向代理、防火墙或其他额外网络防护直接暴露到公网；仅本机使用时应绑定 `127.0.0.1`。所有业务资源继续要求 bearer token。

后台服务和 CLI 必须共享同一套核心逻辑。不得为后台和 CLI 分别实现两套策略、风控或数据解释逻辑。

### 本地前端控制台

本地前端控制台对接同一 HTTP API。准备页提供东方财富候选预览、按名称/代码提交搜索、多选确认、观察池维护和全量替换导入提示，并分别处理未配置、无效 Key、额度、网络、空结果、预览过期和过滤警告。顶级“行情”工作台使用左侧决策标的扫描器和右侧五个详情标签：`概览`、`K 线`、`资金流`、`分时强弱`、`数据引用`；小屏幕将扫描器改为标的选择抽屉。建议和复盘页显式区分后端 current 最新状态与 history 完整周期历史，不在浏览器全量分组或按通知原始建议 ID 猜测关联。设置页提供 SMTP 脱敏配置、测试和失败投递重试。前端不自动真实下单，不控制真实交易客户端，不读取真实券商凭据，也不自行推断证券类型、交易制度或计算技术特征、VWAP、强弱分类、策略、风控和建议。

SMTP 密码和东方财富/妙想 Key 经用户确认后允许明文保存在本地 SQLite。读取 API 和 Web 只能返回脱敏状态，不得回填原值；数据库导出和备份会包含这些凭据，界面必须明确提示。该例外不适用于券商凭据或其他秘密。

## 5. 股票池

首版证券池只包含：

1. 手动持仓台账中的沪深 A 股或 ETF。
2. 用户从东方财富候选或证券搜索结果中确认加入、且 `plan_enabled=true` 的本地观察证券。

当前持仓来自手动持仓台账。持仓证券优先级高于观察证券。远端候选预览和搜索结果不是股票池状态；只有人工确认后的本地记录才可能进入决策。证券类型或交易制度未知时，非持仓标的只能观察，持仓标的保守 `hold` 并要求人工复核。

## 6. 决策节奏

系统采用“收盘计划 + 盘中触发”的节奏。

收盘后：

- 读取手动持仓台账和 `plan_enabled=true` 且证券元数据已验证的观察池列表。
- 在 `15:15` 起检查当日日 K 和报价是否就绪，未就绪时每 5 分钟重试，最晚到 `16:30`。
- 增量更新前复权日 K 和资金流，固化行情输入快照并计算历史特征和市场结构。
- 生成并原子激活下一 `XSHG` 交易日的计划、观察列表、允许动作、仓位约束和失效条件。

盘中：

- 每 3 分钟为当前持仓、当日活动计划标的和 `plan_enabled=true` 且元数据已验证的观察标的统一采集一次报价和当日分钟线，并生成带规则版本的分时强弱。
- 账户估值、活动计划消费、策略、风控、建议和通知仍只处理当前持仓和当日活动计划标的；计划外非持仓观察标的仅展示已保存行情事实。
- 日 K 不在盘中请求 provider。无活动计划时为采集标的复用截至前一交易日可用的不可变历史快照或从本地事实精确物化；有活动计划时，计划内标的必须使用计划冻结的收盘历史引用，计划外采集的仅展示观察标的才使用本地历史路径。冻结引用缺失、不可用或损坏时必须显式失败，不能静默替换为更新快照。
- 报价和分钟数据的 stale 阈值由 `QT_MARKET_STALE_TRADING_MINUTES` 配置，默认 6 个有效交易分钟；午休和非交易时段不计入，实际生效值随输入快照保存。
- `buy/add` 必须通过活动计划、多因子确认和硬性风控；持仓新风险可以覆盖计划。
- 所有建议必须先通过风控规则，再进入推送。

## 7. 核心模块

### Data Adapters

封装外部数据源，包括 AkShare、沪深证券产品目录、东方财富只读自选接口以及后续备用数据源。外部接口变化应尽量只影响 adapter；策略、风控、计划和 Web 只消费内部证券元数据和行情模型。

ETF 代码必须经过所需公开目录和交易所分类交叉验证。结构化长短名称兼容通过后，内部目录使用公开 spot 的较完整展示名称，保留其中的管理人信息以区分同主题 ETF；名称不得用于猜测证券类型、交易所或 T+0/T+1。

### Manual Position Ledger

维护当前真实持仓的手动台账。手动持仓台账是系统计算仓位、成本、止损、止盈和持仓风险的唯一权威来源。

持仓台账至少应包含股票代码、名称、持仓数量、可用数量、成本价、建仓日期、最近更新时间和备注。后台服务只读取持仓台账，CLI 负责手动维护持仓台账。

系统不得从东方财富模拟组合推断真实持仓成本、数量或可用数量。建议中必须保留持仓台账更新时间和行情数据时间。

### Manual Cash Account

维护现金余额、累计转入、累计转出和净本金。手动资金账户是系统计算可用买入资金、总资产、仓位比例和资金约束的唯一权威来源。

初始化本金、模拟银证转入、模拟银证转出和现金校准必须由 CLI 或 API 明确记录。统一后台服务只读取资金账户，不得自动修改现金余额或净本金。

### Account Valuation

账户估值由手动持仓台账、手动资金账户和 AkShare 行情快照共同生成。估值结果应包含现金余额、持仓市值、总资产、浮动盈亏、总盈亏和仓位比例。

当行情部分缺失时，持仓级别可以保留可用报价，但账户级别总市值、总资产和仓位比例不得用不完整覆盖率伪装成完整账户估值。

### Unified Service Runner

统一后台服务通过 `DecisionWorkflow` 按配置节奏读取手动台账、固化行情输入、生成账户快照并写入本地结构化日志。旧账户快照 CLI、HTTP 写入口和调试轮询入口均不得形成独立行情采集路径。它不自动下单、不操作真实交易客户端、不自动修改手动持仓台账或手动资金账户。

### Universe Builder

生成首版证券池，并保留来源标记。来源至少包括 `holding` 和 `watch_pinned`。每个成员引用后端验证的 `exchange`、`instrument_type`、`settlement_cycle`、`price_limit_ratio`、可用时的权威 `listing_date`、`metadata_source` 和检查时间。

### Market Data Cache

保存证券目录和预览缓存、持仓台账快照、行情快照、250 个交易日的前复权日 K、适用品种的 60 个交易日完整资金流、当日 1 分钟线、分时强弱、计划和建议。ETF 资金流为 `not_applicable`。日 K 和适用的资金流事实采用版本化追加及不可变数据集快照；本地日 K 快照只选择 cutoff 向前精确的 `XSHG` 目标交易日和各日最新事实，不使用非交易日或窗口外事实补行，也不自行生成 provider 完整窗口证据。原始分钟线只滚动保留最近 20 个 `XSHG` 交易日，派生强弱、建议输入和审计引用不得随清理删除。

### Decision Workflow

统一编排 `close`、`intraday`、`backfill` 和 `cleanup` 工作流。每轮创建 `run_id` 和 `market_input_snapshot_id`，下游只读取本轮已经固化的输入，不在策略、风控或前端刷新阶段再次请求第三方。工作流必须具有幂等键、原子执行租约、异常终态、短事务边界、逐标的降级和运行指标；HTTP、CLI 与调度器并发命中同一轮时只允许一个执行者调用 provider。

当前 decision intraday 的 run 和市场输入必须共同保存有效交易日及历史 cutoff；正常盘中分别为本交易日和前一 `XSHG` 交易日。旧失败 run 重试时只补齐缺失日期，不覆盖已有非空 provenance，并保证最终 run 与 `MarketInputSnapshot` 的日期一致。

`intraday` 有显式 `decision` 和 `display_only` 两种模式。调度器始终使用 `decision`；非交易时段的 HTTP/CLI 行情刷新必须显式请求 `display_only`。display-only 使用“模式 + 有效交易日 + 请求时刻上海墙钟三分钟 bucket”作为运行身份，并将实际模式、有效交易日、history cutoff、请求标的 scope 和后端计算的 10 分钟 lease 同时保存到 run、MarketInput、响应和审计。它只允许写 Universe、quote/minute/history/strength、逐数据集质量、CaptureRun、MarketInput 和审计，在账户估值、策略、风控、计划、建议及通知前硬退出；即使存在持仓也不得写 AccountSnapshot、TradingPlan、Recommendation、Notification、邮件 outbox 或反馈，`recommendation_ids` 恒为空。

Web 两阶段行情刷新必须先执行 backfill，再执行 intraday，并分别展示运行身份、终态、降级和 warning；两阶段不是跨工作流事务，后阶段失败不得回滚已经完成的日 K。backfill run 保存规范化的请求标的 scope 和 4 小时 lease，intraday run 保存 10 分钟 lease；`409 workflow_in_progress` 的客户端必须按精确 run ID 读取这些后端事实并有界跟随，不得自行猜测 scope、超时或立即显示失败。

display-only 的无源时间报价默认不可用于价格动作。仅在观测严格晚于有效交易日收盘、已固化同日的前复权日 K 存在且其收盘价与报价严格一致时，才能以交易所会话收盘时间标记为可追溯的 partial 报价；午休、价格不一致或同日 K 缺失均不得补造行情时间。

`intraday`、`close` 和 `cleanup` 由后台按交易时段调度，`backfill` 由认证 HTTP 或 CLI 按需运行；四种类型都经过同一 `DecisionWorkflow`，不存在独立的计划或建议写入服务。盘中调度触发时间只确定交易日和三分钟幂等周期，运行开始、结束、获取时间和耗时必须使用真实可推进时钟。每轮监控至少记录交易日、周期、幂等键、起止与耗时、请求/处理标的数、逐数据集 `complete/degraded/failed/stale` 计数、provider 调用与耗时、行数、计划/建议/通知/outbox 数量和安全错误摘要；调度状态另记录 `overrun_count`、`skipped_count` 和 `last_reason`。

### Strategy Engine

生成候选信号。首版采用规则策略，以技术面为主，适用品种的资金流和分时强弱作为确认或过滤条件。ETF 不使用资金流确认，仍必须由日线结构和盘中强弱等至少两个独立有效因子通过买入侧门禁。策略输出必须可解释。

### Risk Engine

在最终建议前执行硬性风控约束。风险约束优先于策略信号。

### Recommendation Engine

合成最终建议，统一字段格式，并确保每条建议都包含理由、风险、失效条件、仓位约束和数据时间戳。

### Notifier

负责输出建议。当前范围支持本地控制台、数据库/Web/API 通知、JSONL 结构化日志和 SMTP outbox。`buy/add/sell/reduce` 进入即时邮件，`hold/watch/avoid` 只进入收盘每日摘要；关键工作流故障先写本地通知、控制台和 JSONL，再在 SMTP 可用时进入系统告警 outbox。邮件达到最大重试次数成为 `dead` 时也必须生成可在 Web 查看且投射到控制台和 JSONL 的本地告警，不能因 SMTP 未配置或故障而丢失告警。邮件投递失败独立重试，不能回滚建议或重跑决策工作流。

### Simulation Operation Layer

预留东方财富模拟组合接口作为自动模拟操作决策层。该层可在后续接收系统建议，并在模拟组合中执行模拟买入、模拟卖出或组合调整，用于演练和复盘策略表现。

模拟操作决策层不得修改手动持仓台账，不得代表真实交易，不得影响真实持仓建议的成本和数量计算。

### Audit Log

记录输入数据、信号、风控结果、最终建议、推送结果和人工反馈。审计日志用于复盘、排错和后续回测。

## 8. 技术栈约束

首版按 Python 项目设计。

- Python 3.11 或更高版本。
- 包管理优先考虑 uv。
- 数据处理优先使用 pandas。
- 数据结构和配置校验优先使用 pydantic。
- 后台调度可使用 APScheduler 或同类成熟调度器。
- 交易日历使用 `exchange-calendars` 的 `XSHG` 日历。
- Web 使用 Node 24、Vue 3、TypeScript、Vue Query 和 Apache ECharts。
- 代码应为后续测试、回放和替换数据源保留边界。

## 9. 成功标准

首版完成后，应能做到：

- 通过 CLI 检查配置、维护持仓台账、预览/搜索/确认观察证券、同步行情数据、显式运行统一收盘工作流、查看计划和建议。
- 通过后台服务在交易日自动执行计划生成和盘中触发检查。
- 后台、CLI 和 HTTP 手动触发共享同一 `DecisionWorkflow` 和可追溯输入。
- 对决策启用标的维护 250 日前复权日 K、适用品种的 60 日资金流和最近 20 个交易日原始分钟线；ETF 明确保存资金流不适用状态。
- 对每条建议给出可解释理由和风险条件。
- 对每条建议保留 `run_id`、行情输入、计划、数据质量、通知和审计引用。
- Web 行情工作台和 SMTP 设置页可以在数据降级时独立工作，且不提供下单控件。
- 在不接入自动交易的情况下支持用户手动决策。
