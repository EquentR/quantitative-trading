# 行情决策工作流与 Web 工作台设计

## 背景

`2026-07-12-csv-market-snapshots-design.md` 已经完成首期可追溯报价快照，但其中明确列为非目标的历史 K 线、资金流、分时强弱、策略消费、收盘计划消费、通知和调度集成仍未落地。当前代码已经有手动持仓台账、手动资金账户、股票池、行情报价快照、收盘计划、策略、风控、建议、通知、审计、调度器和 Web 控制台骨架，但这些能力尚未形成一条使用同批次数据、可降级、可重试且可完整追踪的决策链。

本设计把上述非目标展开为一个综合阶段。阶段目标不是增加更多独立入口，而是让后台常驻服务、CLI、HTTP API 和 Web 工作台共同消费一条统一的后端决策工作流。

- SMTP 密码允许通过 Web 配置并明文保存在本地 SQLite。该决定只适用于邮件投递凭据，不改变禁止读取券商凭据、禁止提交密钥到 git 的安全边界。

若本文与上述阶段文档在这些事项上冲突，以本文为准；未明确替代的既有规约继续有效。

## 目标

- 为决策启用集合保存 250 个交易日的前复权日 K 线。
- 为决策启用集合保存最近 60 个交易日的逐股资金流明细。
- 在交易时段保存当日 1 分钟行情并计算可解释的分时强弱。
- 使用同一批次固化输入完成计划、策略、风控、建议、通知和审计。
- 将收盘计划作为次日 `buy/add` 的硬门禁，同时允许盘中新风险覆盖原计划。
- 打通控制台、Web/API、JSONL 和邮件通知，并支持未读、已读、反馈、重试和去重。
- 提供顶级“行情”Web 工作台，展示 K 线、资金流、分时强弱和数据引用。
- 保持后台常驻服务和 CLI 共享核心逻辑，并记录每轮运行成本与故障状态。

## 非目标

- 不自动下单，不创建真实订单，不控制真实交易客户端。
- 不读取券商账号、密码、token、cookie 或 API key。
- 不从东方财富模拟组合推断真实持仓、成本、数量、可用数量或资金。
- 不扫描全市场，不扩大到未启用计划的非持仓自选股。
- 不把资金流作为独立买卖触发器。
- 不使用黑盒模型作为首版核心策略。
- 不接入 Telegram、飞书、钉钉或短信。
- 不实现 WebSocket、消息中间件或分布式任务系统。
- 不在前端计算技术特征、强弱分类、策略、风控或建议。
- 不实现专业交易终端、画线工具、用户自定义指标或自动策略编辑器。
- 不自动归档日 K 和资金流；本期只对原始分钟线执行滚动清理。

## 总体架构

### 统一应用工作流

新增 `DecisionWorkflow` 应用服务，作为后台调度器和 CLI 的共同入口。HTTP 手动触发接口也只能调用该服务，不得复制采集、策略或风控逻辑。

`DecisionWorkflow` 只负责编排，不承载特征公式或交易规则。标的选择、采集、特征、计划、策略、风控、建议、通知和审计继续由边界清晰的领域 service 实现，并通过内部模型通信，避免形成不可测试的巨型 service。

每轮运行创建唯一 `run_id`，按以下顺序执行：

```text
确定决策启用标的
  -> 单次采集并固化标准化输入
  -> 计算可解释特征
  -> 加载适用的活动收盘计划
  -> 策略生成候选动作
  -> 风控裁决和仓位约束
  -> 生成契约化建议
  -> 创建通知与邮件 outbox
  -> 写入审计和运行指标
```

同一轮的下游阶段只读取已经固化的输入，不在策略、风控或建议阶段临时访问第三方数据源。每轮形成：

```text
run_id
  -> market_input_snapshot_id
  -> plan_id（收盘生成或盘中消费，可为空）
  -> recommendation_id
  -> notification_id
  -> delivery_id
```

所有引用都必须能够正向查询和反向追踪。Web 只读取这些后端事实。

### 工作流类型

`DecisionWorkflow` 支持以下类型：

- `close`：收盘数据就绪检查、日线和资金流增量采集、特征计算、次日计划发布和每日摘要。
- `intraday`：报价和当日分钟数据采集、账户估值、分时强弱、计划消费、策略、风控、建议和即时通知。
- `backfill`：按标的补齐日线或资金流基线，不生成建议。
- `cleanup`：删除超出保留期的原始分钟数据，不删除派生决策快照。

### 幂等和事务边界

工作流幂等键至少包含任务类型、交易日和计划周期；盘中轮次还包含按 3 分钟对齐的周期开始时间。相同幂等键重试时复用已有运行记录，并且不能重复发布计划、建议、通知或邮件。

外部采集不持有长数据库事务。每个数据集先完成 adapter 映射和模型校验，再按明确事务保存。计划发布、建议与本地通知创建、邮件 outbox 创建分别使用能够保证引用完整性的短事务。数据库约束、外键或内部契约失败时，当前业务阶段回滚并将整轮标记为失败。

## 标的范围

重数据和决策只覆盖决策启用集合：

- 手动持仓台账中的所有股票，无条件进入。
- `plan_enabled=true` 的非持仓自选置顶股票。

同一股票同时来自持仓和自选置顶时只采集一次，并保留两种来源，持仓语义优先。`plan_enabled=false` 的非持仓自选仍可在可见股票池中维护，但不得拉取日 K、资金流和分时重数据，也不得进入计划和建议。

盘中轮次只能覆盖当前持仓和当日活动计划中的标的。盘中不得因为 adapter 返回额外股票而扩大范围。

## Adapter 边界

### 内部接口

市场模块提供以下 Protocol：

- `QuoteProvider.get_quotes(symbols)`
- `DailyBarProvider.get_daily_bars(symbol, start_date, end_date, adjustment)`
- `MoneyFlowProvider.get_daily_money_flow(symbol, start_date, end_date)`
- `IntradayProvider.get_minute_bars(symbol, trade_date, interval)`
- `TradingCalendar`：判断交易日、计算下一交易日、返回交易时段。

AkShare 是四类市场数据的首选公开数据源。现有东方财富公开单票行情和腾讯 quote 兜底仍只用于报价 adapter，不自动扩展为历史、资金流或分钟数据兜底。交易日历使用 `exchange-calendars` 的 `XSHG` 日历，并保留经认证 CLI 的人工强制运行和跳过能力。

adapter 必须完成第三方字段映射、数值转换、单位统一、股票代码校验、时间和交易日校验，只返回项目内部模型。核心策略不得 import AkShare、requests 或第三方响应模型。

### 调用成本约束

- 报价按 provider 能力对本轮全部标的批量调用一次。
- 日 K 和资金流首次回填完整基线，收盘时只刷新缺失日期和最近 5 个交易日的校正窗口，不在每个盘中周期或每次前端刷新时重复拉取完整序列。
- 分钟数据只请求当前交易日，并按“股票 + 分钟”幂等写入。
- 同一轮不得由账户估值、策略和前端刷新分别重复调用 provider。
- 每轮记录 provider 调用次数、耗时、标的数、返回行数、写入行数和失败数。

## 标准化数据模型与存储

### 运行和逐标的结果

新增 `market_capture_runs`：

- `id/run_id`
- `workflow_type`
- `trade_date`
- `period_start/period_end`
- `idempotency_key`
- `status`：`running/succeeded/degraded/failed`
- `started_at/finished_at`
- `requested_symbols/processed_symbols`
- `provider_calls/rows_received/rows_written`
- `warning_count/failure_count`
- `error_summary`

新增 `market_capture_results`，每个 `run_id + symbol + dataset` 一条结果：

- `dataset`：`quote/daily_bar/money_flow/minute_bar/intraday_strength`
- `status`：`complete/degraded/failed/stale`
- `data_start/data_end/data_time/fetched_at`
- `expected_rows/actual_rows`
- `source`
- `warning/error_summary`

失败结果必须持久化，不能用空映射或空值冒充成功。错误只保存经过现有 sanitization 处理的摘要。

运行状态语义固定如下：全部必需数据完成且没有降级时为 `succeeded`；至少一个标的仍有可用必需数据，但存在失败、stale 或可选数据缺口时为 `degraded`；内部完整性错误或所有请求标的都没有可用必需数据时为 `failed`。空决策集合不调用 provider，保存空快照并以带 warning 的 `succeeded` 结束。

### 报价快照扩充

继续复用版本化 `quote_snapshots`，并扩充内部 `QuoteSnapshot` 的可选标准字段：

- 前收、开盘、最高、最低、最新价和涨跌幅。
- 成交量和成交额。
- `trading_status`：`normal/suspended/unknown`。
- `limit_status`：`none/up/down/unknown`。
- 市场数据时间、获取时间、来源、质量状态和 warning。

adapter 无法可靠映射的可选字段保持 `None/unknown` 并使报价为 partial，不得推测停牌或涨跌停状态。现有只依赖最新价的账户估值仍可消费 partial 报价，但策略按字段级质量决定可用性。

### 日 K

`daily_bars` 保存：

- `id/content_hash`
- `symbol/trade_date`
- `adjustment=forward`
- `open/high/low/close`
- `volume/amount`
- `source/source_updated_at/fetched_at`

日 K 使用追加版本而不是覆盖旧值。唯一键为 `symbol + trade_date + adjustment + content_hash`；provider 对历史值发生修正时新增版本，当前查询选择最新版本。`content_hash` 由标准化字段的规范 JSON 计算 SHA-256，不包含抓取时间等非业务字段。

每次收盘或回填创建不可变 `history_snapshots`，记录 `run_id`、标的、复权方式、覆盖区间、行数、内容摘要和数据质量；`history_snapshot_members` 按时间顺序引用本次实际使用的 `daily_bars.id`。`MarketInputSnapshot.history_snapshot_refs` 只能引用该快照，因而后续数据修正不会改变历史计划所看到的序列。

首次启用标的时补齐最近 250 个交易日，之后按交易日增量采集。所有策略和图表默认只消费前复权序列；未经复权或后复权数据不得混入同一计算窗口。

日 K 数据在本期不自动删除。API 默认最多返回最近 250 个交易日。

### 资金流

`daily_money_flows` 保存：

- `id/content_hash`
- `symbol/trade_date`
- 主力净流入额和净占比
- 超大单净流入额和净占比
- 大单净流入额和净占比
- 中单净流入额和净占比
- 小单净流入额和净占比
- `source/source_updated_at/fetched_at`

金额统一使用人民币元，占比统一使用百分点数值。资金流同样使用追加版本，唯一键为 `symbol + trade_date + content_hash`，当前查询选择最新版本。

每次收盘或回填创建不可变 `money_flow_snapshots` 和对应 members，记录本次实际使用的行版本、覆盖区间、内容摘要和数据质量。`MarketInputSnapshot.money_flow_snapshot_refs` 只能引用该快照。

首次启用标的时补齐最近 60 个交易日，之后增量采集。资金流事实版本和快照在本期不自动删除，API 默认最多返回最近 60 个交易日。

### 分钟线

`minute_bars` 保存：

- `symbol/trade_date/minute`
- `open/high/low/close`
- `volume/amount`
- `source/source_updated_at/fetched_at`

`minute` 必须位于 A 股交易时段，且带 `Asia/Shanghai` 时区语义。唯一键为 `symbol + minute`。每轮对当日已返回分钟做幂等更新，不重复插入相同分钟。

原始分钟线滚动保留最近 20 个交易日。清理边界通过 `TradingCalendar` 计算，不使用自然日近似。删除原始分钟线前不删除引用它们生成的强弱快照、建议输入和审计记录。

### 分时强弱快照

`intraday_strength_snapshots` 保存本轮计算值，而不是只保存最终标签：

- 当前价相对前收的变化。
- 当前价相对当日开盘价的变化。
- 当前价相对 VWAP 的变化。
- 5 分钟动量。
- 15 分钟动量。
- 当日价格区间位置。
- 最近 5 分钟相对前 20 分钟的分钟量比。
- 每项的状态、贡献方向和人类可读理由。
- 最终 `strong/neutral/weak`、置信度和降级原因。
- 使用的最后分钟、数据覆盖率、数据源和计算规则版本。

VWAP 使用当日累计成交额除以累计成交股数。若 provider 的成交量单位不是股，adapter 必须先统一单位。日内位置为 `(current-low)/(high-low)`；当分母为零时该组件不可用，不得填入伪造值。

前收优先使用当日 quote 的标准字段；缺失时可以使用前一交易日的最新有效前复权日线收盘价。开盘、高低价优先从当日有效分钟序列计算。任何回退都必须记录来源，且日线日期必须与交易日历严格相邻。

首版分类使用可配置、版本化的规则，不使用黑盒评分。百分比变化 `x` 在 `x >= threshold` 时贡献 `+1`，在 `x <= -threshold` 时贡献 `-1`，否则贡献 `0`。默认阈值为：前收 `0.5%`、开盘 `0.3%`、VWAP `0.2%`、5 分钟动量 `0.3%`、15 分钟动量 `0.6%`；日内位置大于等于 `0.70` 贡献 `+1`，小于等于 `0.30` 贡献 `-1`。

最近 5 个完整分钟的平均成交量除以前 20 个完整分钟的平均成交量得到分钟量比；不足 25 个有效分钟或分母为零时不可用。量比不贡献方向，只调整置信度。六个方向组件至少四个可用，方向和大于等于 `3` 且当前价不低于 VWAP 时分类为 `strong`；方向和小于等于 `-3` 且当前价不高于 VWAP 时分类为 `weak`；其余为 `neutral`。`strong/weak` 基础置信度为 `medium`：量比大于等于 `1.5` 且 5 分钟动量同向时提升为 `high`，量比低于 `0.8` 或不可用时降为 `low`。`neutral` 为 `low`。组件不足四个时状态为降级，不能为 `buy/add` 提供完整分时确认。每次快照必须保存实际阈值和规则版本，以便后续调整规则后仍能解释历史结果。

### 市场输入快照

扩充现有 `MarketInputSnapshot` 和 repository 校验：

- `universe_snapshot_id`
- `quote_snapshot_refs`
- `history_snapshot_refs`
- `money_flow_snapshot_refs`
- `intraday_strength_snapshot_refs`
- 每个标的、每个数据集的质量状态和覆盖范围
- `data_time/fetched_at`
- `warnings`
- `capture_run_id`

日线和资金流引用指向不可变的数据集快照及其事实版本成员，包含覆盖区间、内容摘要和计算使用的最后交易日；分钟决策引用指向不可变强弱快照。分钟缓存允许幂等更新并按保留期清理，但建议解释所需的派生值、规则版本、质量状态和数据摘要必须随决策快照长期保留。

盘中快照的聚合 `data_time` 取全部有效报价和分时强弱数据时间的最早值，避免单条较新数据掩盖同轮较旧数据；收盘快照的聚合 `data_time` 为当日收盘时间，前提是硬性数据均已更新到该交易日。每个数据集仍保留自己的起止时间，聚合时间不能替代逐数据集质量检查。

## 历史特征和收盘计划

### 日线特征

收盘工作流在后端计算并保存：

- MA5、MA10、MA20、MA60。
- 5、10、20 和 60 个交易日收益。
- MA5、MA10 和 MA20 的斜率方向。
- 20 日最高、最低和当前区间位置。
- ATR14。
- 5 日平均成交量、20 日平均成交量和当日量比。

不足相应窗口的数据特征必须标记不可用。MA60 或 60 日收益缺失可以使计划降级，但日线不足 20 个有效交易日时不能为非持仓标的生成买入观察计划。

### 市场结构价位

现有按手动持仓成本机械推导支撑、压力和止损的占位逻辑在本期移除。首版市场结构价位从以下候选中选择并保存选择理由：

- 当前价下方最近的 MA5、MA10、MA20 和最近 5/10/20 日低点作为支撑候选。
- 当前价上方最近的最近 5/10/20 日高点作为压力候选。
- ATR14 用于判断价格缓冲和失效区间，不用于承诺目标收益。

手动成本仍用于持仓盈亏、风险和仓位判断，但不能冒充市场支撑。若没有有效市场结构候选，相关价位为空并使计划降级，不得回退到伪造的成本比例价位。

### 计划模型和发布

每个计划包含：

- `plan_id/version/status`
- `source_run_id/market_input_snapshot_id`
- `created_at/data_time`
- `applicable_trade_date/expires_at`
- 计划级数据质量和 warnings
- 每个标的的来源、持仓上下文和账户上下文引用
- 趋势、量价和资金流判断及逐项理由
- 机器可执行的价格、强弱和量能条件
- 允许动作和禁止动作
- 建议仓位区间和硬上限
- 风险、失效条件和数据依赖

计划状态为 `draft/active/superseded/expired/stale`。`stale` 表示计划在正常到期前因关键数据或台账变化失去可执行性。同一适用交易日只能有一个活动版本。新版本原子激活时将旧版本标为 `superseded`。计划默认适用于下一个 `XSHG` 交易日，并在该交易日 `15:00 Asia/Shanghai` 到期。

行情和日 K 是发布计划的硬前提。资金流缺失允许发布降级计划，但计划必须注明资金确认不可用。收盘窗口结束仍不满足硬前提时不发布半成品计划。

## 盘中计划消费、策略和风控

### 计划门禁

`buy/add` 必须同时满足：

- 存在适用于当日且状态为 `active` 的计划。
- 标的在计划内且计划允许该动作。
- 命中计划内机器可执行的价格、趋势或量能条件。
- 获得日线结构和盘中强弱等至少两个互相独立的有效因子确认。
- 通过所有硬性风控。

资金流只能增加确认或作为过滤条件，不能独立触发动作。资金流缺失时移除该确认并降低置信度；若剩余有效因子不足，动作降级为 `watch`。

计划外出现的新机会只能输出 `watch`，不能临时升级为 `buy/add`。无活动计划时，非持仓标的只能 `watch/avoid`。

### 持仓风险覆盖

持仓风险管理不受买入侧计划门禁阻断。盘中新出现的止损、回撤、跌破结构、仓位超限或其他硬性风险可以输出 `sell/reduce/avoid` 并覆盖原计划。建议必须记录原计划条件、覆盖原因和最终风控裁决。

若风险动作所依赖的当前价、成本、数量或可用数量缺失，则不得猜测动作或数量，必须降级为保守 `hold` 并明确要求人工复核。可用数量为零时，`sell/reduce` 按既有规约降级为 `hold`。

### 动作语义

- `buy`：未持仓标的满足活动计划和多因子条件后的新建仓位建议。
- `sell`：持仓风险或计划失效要求退出，且可用数量和风控允许。
- `add`：已有持仓满足活动计划、确认条件和仓位约束后的加仓建议。
- `reduce`：降低已有持仓风险或兑现计划条件，且可用数量允许。
- `hold`：持仓条件仍成立，或数据不足以支持可执行动作并需要保守管理。
- `watch`：条件接近、计划外机会、数据降级或尚未通过全部买入条件。
- `avoid`：风险、流动性、数据质量或市场结构明确不适合参与。

策略先输出带机器原因和人类原因的候选动作。风控可以否决、降级、收紧数量和仓位，也可以因新风险生成持仓侧动作；风控不得把弱策略信号升级为 `buy/add`。

### 台账和账户一致性

每轮盘中策略前重新读取最新手动持仓和手动资金，不沿用计划中的推算数量。若台账自计划生成后发生变化，使用最新台账重新计算数量、成本上下文和仓位；无法重算时降级。

手动台账和手动资金仍是唯一权威源。人工执行反馈不修改台账、现金或账户估值。A 股交易单位、T+1 可用数量、单票仓位、总仓位、单日新增买入、单日交易次数、连续亏损冷却和流动性规则继续使用 `docs/trading-policy.md` 的可配置默认值。

## 建议契约和去重

每条最终建议必须保存：

- `recommendation_id/action/confidence`
- `run_id/market_input_snapshot_id/plan_id`
- 数据时间、获取时间和建议时间
- 策略原始动作、机器原因和人类原因
- 风控裁决、拒绝或降级原因
- 风险和 `invalid_if`
- 建议数量、仓位区间和实际生效上限
- 手动持仓台账、手动资金和账户估值引用
- 日 K、资金流、分时强弱引用或明确缺失状态
- 数据质量、warnings 和审计引用

`buy/add/hold/watch` 必须有非空失效条件；其他动作也必须包含适用风险和失效语义。最终动作只表示系统建议，不表示真实交易已执行。

建议去重键使用交易日、3 分钟周期、标的、最终动作、计划版本和条件指纹。条件指纹覆盖触发规则、风险、失效条件和仓位约束。完全相同条件的重跑复用原建议；动作、风险、失效条件、数量或计划版本发生实质变化时允许生成新建议。

## 数据质量与降级矩阵

| 缺失或异常 | 持仓标的 | 非持仓标的 |
|---|---|---|
| 当前报价缺失或过期 | 最后成功价只用于标记 stale 展示；暂停价格触发型动作，输出保守 `hold` 和人工复核 | 禁止 `buy/add`，只能 `watch/avoid` |
| 日 K 不足或日期不符 | 不生成市场结构价位，只保留不依赖日 K 的持仓管理并降级 | 不进入买入观察计划 |
| 资金流缺失 | 去掉资金确认并降低置信度，剩余因子不足时降级 | 同左，资金流不得独立触发 |
| 分时数据缺失、覆盖不足或延迟 | 取消分时确认，依赖分时的即时动作暂停或降级 | 禁止 `buy/add` |
| 账户估值部分可用或不可用 | 禁止 `buy/add`，持仓建议说明仓位信息缺口 | 禁止 `buy/add` |
| 台账关键字段缺失 | 不输出依赖成本、数量或可用数量的动作，转人工复核 | 不适用 |
| 活动计划缺失或过期 | 仍可做持仓风险管理 | 只能 `watch/avoid` |
| 明确停牌或不可交易 | 不输出假定可成交的动作，保守 `hold` 并说明状态 | `avoid` |
| 明确涨跌停 | `sell/reduce` 可以保留为风险建议，但必须说明可能无法成交；禁止新增买入 | 禁止 `buy/add`，只能 `watch/avoid` |

盘中报价和分钟数据默认在最新数据时间落后预期周期超过 6 个有效交易分钟时判定 stale，午休和非交易时段不计入；该阈值可配置，实际生效值必须保存在快照。收盘日 K 必须包含当前交易日才算 ready。任何旧数据用于展示时必须同时显示原始数据时间和 stale 标记，不能用系统当前时间伪装为新数据。

逐标的外部失败不会终止其他标的。数据库写入、外键引用、交易日错位、模型契约或内部不变量错误会终止当前整轮并告警。

## 调度设计

### 交易时段

后台调度使用 `TradingCalendar`，默认只在 `XSHG` 交易日运行：

- 盘中工作流：`09:30-11:30` 和 `13:00-15:00 Asia/Shanghai`，以每个交易时段起点为基准每 3 分钟一次。
- 收盘工作流：`15:15` 首次运行；硬前提未就绪时每 5 分钟重试，最晚到 `16:30`。
- 原始分钟线清理：每个交易日 `16:35` 独立执行，不依赖收盘计划是否发布；失败进入单独告警并在下一次维护周期重试。

盘中工作流每轮统一完成一次报价/分钟采集、账户估值、强弱、策略、风控、建议和通知。现有每分钟 intraday cron 和独立重复采集账户行情的调度方式必须收口，不能与新工作流并行保留。

### 并发、错过和恢复

- 每种工作流 `max_instances=1`。
- 上一盘中轮次仍未结束时不启动并发轮次，记录 `overrun/skipped` 指标并告警。
- 调度器合并错过的盘中触发，只补当前仍有效的周期，不回放过期的每个 3 分钟周期。
- 服务在 `15:15-16:30` 内重启且当日计划未发布时，立即进入收盘就绪检查。
- 超过 `16:30` 后不自动绕过硬截止时间；通过 CLI 显式补跑并记录人工原因。
- 调度状态持久化，HTTP 服务不能因任务异常退出。
- 人工强制运行、跳过日历或晚于截止时间补跑都必须经过认证并写入审计。

## 通知、邮件与审计

### 本地通知闭环

最终建议先写数据库通知，再投射到：

- 本地控制台摘要。
- Web/API 通知列表。
- JSONL 结构化日志。
- 邮件 outbox。

通知状态继续使用 `unread/read/feedback_recorded`。新通知默认 `unread`；标记已读只改变本地处理状态；记录人工执行反馈可以将关联通知标记为 `feedback_recorded`，但不能修改建议、持仓台账或资金账户。

通知去重键包含交易日、标的、动作、计划版本和条件指纹。相同条件不重复通知；风险、失效条件、仓位、动作或计划版本变化时允许创建新通知。

### 邮件节奏

- `buy/add/sell/reduce` 在建议和本地通知成功后立即进入邮件 outbox。
- 关键工作流或数据完整性故障立即进入系统告警邮件 outbox。
- `hold/watch/avoid` 不在每个盘中周期发邮件，只在收盘计划成功发布后形成一封每日摘要。
- 同一活动计划版本的每日摘要只创建一次。

“立即”表示创建 outbox 时不设置业务延迟；实际 SMTP 投递由独立 worker 执行。邮件失败不得回滚或重跑决策工作流。

### 邮件 outbox

邮件投递状态为：

- `pending`
- `sending`
- `retry`
- `sent`
- `dead`

outbox 保存通知引用、收件人、主题、渲染所需的非秘密 payload、尝试次数、下次尝试时间、最后安全错误摘要和发送时间。默认在首次失败后的 1、5、15、30、60 分钟重试，连同首次投递最多 6 次；参数可配置。达到上限后标为 `dead`，生成本地告警。CLI 和 Web 可显式重置为待发送，操作写入审计。

worker 必须使用原子 claim 防止同一邮件被多个执行器同时投递。进程在 `sending` 状态中断时，超过租约的邮件可以恢复为 `retry`。

邮件 worker 在服务运行期间每 15 秒轮询一次，不受交易时段限制，且 `max_instances=1`。轮询只 claim 已到 `next_attempt_at` 的有限批次，避免邮件积压占用决策工作流线程。

首次失败后的重试间隔和发送租约通过 `QT_EMAIL_RETRY_DELAYS_MINUTES`（JSON 数组，默认 `[1,5,15,30,60]`）与 `QT_EMAIL_LEASE_SECONDS`（默认 `60`）配置，并将实际生效值用于 outbox worker。

### SMTP Web 配置

设置页支持：

- SMTP 主机和端口。
- 用户名和密码。
- 发件人和收件人。
- `none/starttls/ssl` 连接模式。
- 启用状态。
- 连接测试和测试邮件。

SMTP 配置保存在本地 SQLite，密码按用户确认明文保存。读取接口只能返回 `password_configured`，不得返回密码。密码输入留空表示保留原值；替换和清除是明确操作。配置变更和测试行为写入审计，但请求体、日志、审计、通知、错误和 outbox 不得包含密码。

设置页必须提示数据库导出和备份会包含邮件凭据。数据库、备份和本地运行数据保持 git 忽略，并限制为本机用户可读。SMTP 未配置或禁用时，本地通知链正常运行，并明确展示邮件通道状态。

## HTTP API

在现有 `/api/v1` 和认证体系下新增或补齐以下资源，不引入第二套业务逻辑：

### 行情和数据引用

- `GET /market/symbols/{symbol}/overview`：标的概览。
- `GET /market/symbols/{symbol}/daily-bars`：最多 250 日 K 线。

- `GET /market/runs` 和 `GET /market/runs/{run_id}`：capture run 和逐标的结果。
- `GET /market/snapshots/{snapshot_id}/trace`：market input snapshot 及其引用解析；现有快照创建和读取接口继续保留。

### 计划、建议、通知和审计

- `GET /plans`、`GET /plans/latest` 和 `GET /plans/{plan_id}`：计划列表、活动版本和详情。

- 现有 `/feedback`：人工执行反馈。
- `GET /audit` 和 `GET /audit/{audit_id}`：审计日志及关联资源。

当前 Web 已经查询但后端尚未稳定挂载的通知和审计路由必须正式纳入应用。

### 调度和 SMTP

- 现有 `/service/status` 和 `/service/scheduler/*` 扩充为统一工作流调度状态和启停。
- `POST /service/workflows/{workflow_type}/run`：经认证手动触发。
- `GET/PUT /settings/notifications/email` 和 `DELETE /settings/notifications/email/password`：SMTP 脱敏读取、更新和密码清除。
- `POST /settings/notifications/email/test`：连接和测试邮件。
- `GET /notifications/email-deliveries`、`GET /notifications/email-deliveries/{delivery_id}` 和 `POST /notifications/email-deliveries/{delivery_id}/retry`：outbox 查询和显式重试。

列表统一分页并提供稳定排序。时间序列接口限制最大窗口。API 只返回后端计算好的特征、分类、理由和质量，不让前端补算策略。所有写操作沿用现有认证、统一错误契约和审计规则。

## CLI

CLI 与后台使用同一 repository、adapter 和应用服务，至少提供：

- 日 K 和资金流基线回填。
- 手动运行一次收盘工作流。
- 手动运行一次盘中工作流。
- 按交易日显式补跑收盘计划。
- 查看工作流、调度和数据质量状态。
- 清理超期原始分钟线。
- 查看、测试 SMTP 配置状态，不在输出中显示密码。
- 查看并重试失败邮件。
- 查看通知和标记已读。

命令提供人类可读摘要和可选 JSON 输出。错误输出不得包含第三方原始响应、数据库路径或凭据。

## Web 行情决策工作台

### 信息架构

新增顶级“行情”导航和独立路由。页面采用高密度、工作型布局：

- 左侧决策标的扫描器。
- 右侧当前标的详情区。
- 桌面端保持同时可见；小屏幕下扫描器变为标的选择抽屉。

扫描器展示：

- 股票代码和名称。
- 持仓/自选来源。
- 最新价和涨跌幅。
- 最新建议动作。
- 分时强弱。
- 活动计划状态。
- 数据质量和未读状态。

支持按动作、持仓来源和异常状态筛选。排序和筛选只处理后端返回的当前股票池，不触发全市场查询。

### 标的详情

右侧提供五个标签：

1. `概览`：持仓/计划/建议摘要、市场结构、强弱组件、风险和数据质量。
2. `K 线`：250 日前复权蜡烛图、MA5/10/20/60 和成交量。
3. `资金流`：60 日主力净额柱、主力净占比折线，超大/大/中/小单图例切换和完整表格。
4. `分时强弱`：当日价格、前收、VWAP、分钟成交量、强弱组件和建议发生点。
5. `数据引用`：`run_id/snapshot_id/plan_id/recommendation_id`、数据时间、来源、质量和 warnings。

建议、复盘、通知和审计页面继续保留，并通过这些 ID 与行情工作台互相跳转。

### 图表范围

图表使用 Apache ECharts：

- 支持 tooltip。
- 支持预设时间窗口和有限 data zoom。
- 对数据缺口、stale 区间和降级状态提供明确标记。
- 分时图标记建议发生时间，但不提供交易按钮。
- 不提供画线工具、自定义指标、复杂叠加或专业终端布局。

图表容器使用稳定宽高或宽高比。长股票名、长 warning 和长字段不得覆盖图表或其他控件。移动端详情标签允许横向切换，图表保持最小可读高度。

### 数据刷新和状态

首版继续使用 Vue Query 定时刷新，不使用 WebSocket。盘中行情刷新节奏不得快于后台 3 分钟采集节奏；通知和工作流状态可以使用独立的较短轮询。页面必须覆盖加载、空数据、部分可用、stale、失败和认证过期状态。

### 设置页

设置页增加 SMTP 配置、通道状态、连接测试、测试邮件和失败投递管理。密码字段永不由 API 回填；页面只显示是否已经配置。页面必须显示明文数据库存储的风险提示。

## 服务绑定与安全边界

后台 HTTP 服务按用户要求默认监听 `0.0.0.0`，便于局域网访问。所有业务读写接口继续使用现有认证；文档必须说明该服务不应未经额外网络防护直接暴露到公网。

继续执行以下边界：

- 手动持仓和手动资金是唯一权威源。
- 外部 adapter 只能提供公开市场数据。
- Web 人工反馈不代表系统确认真实成交。
- 不存在真实交易按钮、订单模型或交易客户端控制路径。
- 建议必须包含数据时间、理由、风险、失效条件和仓位约束。
- 不输出保证收益或确定性收益表述。
- 真实持仓、真实资金、真实反馈、数据库、备份、SMTP 密码和认证 token 不得进入 git 或测试夹具。

SMTP 密码明文入库是本设计唯一新增的秘密存储例外。API 永不回显原值，应用日志和审计不得记录原值，开发和测试不得读取用户真实 SMTP 密码。

## 可观测性

每轮工作流记录并在监控页展示：

- 工作流类型、交易日、周期和幂等键。
- 开始、结束和耗时。
- 标的数和逐数据集成功/降级/失败数。
- provider 调用次数和耗时。
- 返回、写入和清理行数。
- 计划、建议、通知和邮件 outbox 数量。
- overrun、skipped、retry 和最终错误摘要。

结构化日志和审计记录使用关联 ID 串联，但不得包含第三方原始 payload、SMTP 密码或本地敏感路径。

## 测试设计

### 模型、adapter 和特征

- 第三方字段映射、单位、前复权标识、时区和股票代码校验。
- 非有限数值、日期错位、空表、稀疏返回和逐标的失败。
- MA、收益、斜率、ATR14、区间位置和量比边界。
- VWAP、5/15 分钟动量、日内位置、分钟量比和强弱分类边界。
- 不足窗口、零成交量、相同高低价、午休缺口和 stale 数据。
- 每个派生快照保存规则版本和实际阈值。

### Repository 和 migration

- migration 可重复执行并创建约束与索引。
- 日线、资金流和分钟线唯一键支持幂等重跑。
- 逐标的失败可保存和查询。
- 快照引用只能指向匹配标的和数据集。
- 事务错误不会留下半个活动计划或孤立建议。
- 分钟清理严格保留最近 20 个交易日，并保留派生快照。

### 策略、风控和建议

- 七种动作均有确定性 fixture。
- `buy/add` 需要活动计划和多因子确认。
- 计划外机会只能 `watch`。
- 资金流不能单独触发动作。
- 持仓新风险可以覆盖计划。
- T+1、现金、账户估值、单票/总仓位、交易次数和冷却规则。
- 所有降级矩阵分支。
- 建议必填字段、数据引用、风险和失效条件。
- 相同周期重跑不重复建议，条件实质变化允许新建议。

### 工作流和调度

- 250 日 K 线和 60 日资金流首次回填与增量更新。
- 收盘数据未就绪时每 5 分钟重试，成功后只发布一次。
- 收盘硬截止时不发布半成品计划。
- 盘中一轮只执行一次共享采集，不由下游重复请求。
- 单标的外部失败继续处理其他标的。
- 数据库或契约失败终止并回滚整轮。
- 可控时钟覆盖交易日、午休、收盘窗口、重启恢复、错过任务合并和禁止并发。
- 记录 provider 成本、耗时和 overrun。

### 通知和邮件

- 通知未读、已读和反馈状态。
- 条件指纹去重和实质变化重发。
- 动作即时邮件与收盘每日摘要。
- outbox 原子 claim、租约恢复、退避、`dead` 和人工重试。
- SMTP 失败不回滚建议，也不重复运行决策工作流。
- SMTP 密码可以写入数据库，但任何读取 API、日志、审计和错误响应都不包含原值。
- SMTP 未配置时本地通知正常。

### API、CLI 和 Web

- API 认证、分页、最大时间窗口和稳定错误码。
- 通知和审计路由实际挂载。
- 手动触发和晚间补跑写审计。
- CLI 人类摘要与 JSON 输出语义一致。
- Vue 单元测试和 MSW 契约测试覆盖所有页面状态。
- Playwright 覆盖桌面和移动端扫描器、五个标签、SMTP 配置和跨页面追踪。
- ECharts 在桌面和移动视口都通过非空像素检查，标签、图例、tooltip 和长文本不发生不合理重叠。

核心自动测试使用 fake adapter、可控时钟、临时 SQLite 和假 SMTP server，不依赖真实 AkShare 或真实邮箱。真实数据 smoke test 只能通过显式人工命令运行，不能成为默认 CI 前提。

## 验收标准

- 后台常驻服务、CLI 和 HTTP 手动触发共享同一 `DecisionWorkflow`。
- 每个决策启用标的可以回填并查询 250 个交易日的前复权日 K。
- 每个决策启用标的可以回填并查询 60 个交易日的完整资金流拆分。
- 盘中每 3 分钟只采集一次本轮报价和当日分钟数据，并生成带完整组件理由的强弱快照。
- 原始分钟线只保留最近 20 个交易日，历史建议解释不因清理失效。
- 收盘计划在数据就绪窗口内幂等发布，并引用实际输入数据。
- `buy/add` 不能绕过活动计划、多因子确认或硬性风控。
- 计划外非持仓机会只能为 `watch`，持仓新风险仍可覆盖计划。
- 每条建议能追溯到台账、账户、行情、历史、资金流、分时和计划的实际输入或明确缺失状态。
- 控制台、Web/API 和 JSONL 实时接收本地通知，状态可读、可标记和可反馈。
- `buy/add/sell/reduce` 进入即时邮件；`hold/watch/avoid` 只进入收盘每日摘要。
- SMTP 可从 Web 配置和测试，密码可明文入库但永不经 API 回显。
- 邮件失败独立重试，不影响建议工作流成功状态。
- Web 顶级“行情”工作台可以查看概览、K 线、资金流、分时强弱和数据引用。
- 前端不计算策略、风控或建议，不存在自动下单入口。
- 调度每轮记录耗时、调用数、标的数、行数和失败数；3 分钟轮次不会并发叠加。
- 默认自动测试不访问真实市场或邮箱，并且 Python、前端单元测试、构建和关键 Playwright 用例通过。

## 后续计划拆分原则

本文覆盖一个端到端产品阶段，但实施计划必须拆成足够细的、可独立测试和审查的任务。建议按以下依赖顺序分批：

1. 交易日历、数据模型和 migration。
2. 日 K adapter、存储、回填和 API。
3. 资金流 adapter、存储、回填和 API。
4. 分钟数据、保留期、强弱特征和 API。
5. 统一工作流、输入快照和运行指标。
6. 收盘特征、市场结构价位和计划发布。
7. 盘中计划消费、策略、风控和建议。
8. 调度收口、恢复和手动入口。
9. 通知/审计稳定 API 和本地闭环。
10. SMTP Web 配置、outbox、摘要和重试。
11. Web 行情工作台和三个图表。
12. 跨页面追踪、响应式 E2E 和完整回归。

每一批还应继续拆成模型/测试、repository、service、入口、文档和回归等小任务；不得将上述十二项直接当作十二个单次大改。
