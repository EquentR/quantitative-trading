# 行情页盘中展示采集修复设计

## 背景与已复现问题

2026-07-16 盘中，行情页只能看到失败或不可用状态，后台盘中任务连续显示 `0ms`。现场数据库和只读 API 检查确认：

- 手动持仓为空，当日没有活动收盘计划，但存在 4 个 `plan_enabled=true` 的观察标的。
- 现有盘中工作流只把“持仓 + 当日活动计划标的”作为采集范围，因此每轮均为 `requested_symbols=0`、`provider_calls=0`，观察标的没有获得当日行情。
- 调度器把同一个冻结的 `task_now` 注入工作流，导致运行开始和结束时间完全相同，监控固定显示 `duration_ms=0`。
- 日 K 和资金流尚未完成首次回填，页面只能读取旧报价和旧分钟数据，旧质量结果被正确标记为 stale 或 failed，但界面没有区分“尚未首次回填”和“本轮盘中采集失败”。
- AkShare 当前报价和日 K 接口可用；其东方财富 1 分钟接口发生连接中断；AkShare 的新浪 1 分钟接口对 A 股和 ETF 均可返回标准分钟数据。

这些现象不是同一个错误。修复必须同时处理采集范围、运行时钟、分钟数据源降级和页面空状态，同时保持活动计划和硬性风控门禁不变。

## 规约变更

本设计定向扩展以下规约：

- `docs/project-spec.md`
- `docs/trading-policy.md`
- `docs/data-sources.md`
- `docs/recommendation-contract.md`
- `docs/superpowers/specs/2026-07-13-market-decision-workflow-design.md`

原规约中的“盘中范围进一步收窄为当前持仓和当日活动计划内标的”拆分为两个范围：

- 盘中展示采集范围：当前持仓、当日活动计划标的，以及 `plan_enabled=true` 且元数据已验证的观察标的。
- 盘中决策范围：仍然只有当前持仓和当日活动计划标的。

观察标的在没有活动计划时只能获得报价、分钟线、分时强弱和质量状态，不得进入账户估值、策略、风控、建议或通知，也不得生成 `buy/add`。前端刷新仍只读取已保存事实，不直接调用第三方 provider。

## 目标

- 没有活动计划时，已启用观察标的仍能在行情页查看当日报价、分钟图和分时强弱。
- 保持计划外非持仓标的只能展示，绝不生成买入或加仓建议。
- 运行开始、结束和耗时反映真实执行时间，不再因冻结时钟显示 `0ms`。
- 东方财富分钟接口不可用时，通过同一 AkShare adapter 边界降级到新浪分钟接口。
- 日 K 或资金流尚未首次回填时，页面显示明确的中性空状态，不冒充盘中采集故障。
- 保持逐标的降级；展示标的失败不得阻断持仓风险管理或其他标的。

## 非目标

- 不自动下单，不控制真实交易客户端，不读取券商凭据。
- 不放宽活动计划、多因子确认或硬性风控门禁。
- 不在盘中重复采集日 K 或资金流重数据。
- 不新增独立的页面轮询 provider、WebSocket 或第二套行情工作流。
- 不自动补造上一交易日收盘计划。
- 不改变手动持仓台账和手动资金账户的权威性。

## 方案比较

### 方案一：统一盘中工作流内使用双范围

同一 `DecisionWorkflow` 计算展示采集范围和决策范围。报价、分钟线和强弱覆盖展示采集范围，决策链只消费决策范围。

优点是复用现有幂等、租约、adapter、快照、质量和审计边界，不产生重复 provider 调用；缺点是工作流需要明确维护两个集合并通过测试防止越权消费。

### 方案二：新增页面行情刷新任务

单独为 Web 页面采集观察标的行情。实现表面简单，但会形成第二条 provider 调用和写入路径，破坏后台、CLI、HTTP 共享核心逻辑的要求，也更难保证幂等和质量一致性。

### 方案三：自动生成或补跑上一交易日计划

用计划把观察标的带入现有盘中范围。盘中无法用当前报价严格验证上一交易日收盘价，自动补造计划会削弱计划契约并混淆展示与决策目的。

采用方案一。

## 总体架构

盘中工作流在取得全局租约后构建当前统一股票池，并派生两个集合：

```text
当前统一股票池
  -> capture_symbols = 持仓 + 活动计划 + 已启用且已验证的观察标的
  -> decision_symbols = 持仓 + 活动计划

capture_symbols
  -> 报价
  -> 当日 1 分钟线
  -> 分时强弱
  -> 逐标的数据质量
  -> MarketInputSnapshot

decision_symbols
  -> 账户估值
  -> 活动计划条件
  -> 策略
  -> 风控
  -> 建议、通知和审计
```

`capture_symbols` 是本轮实际请求和监控计数的口径。`decision_symbols` 是允许进入交易决策链的唯一口径。实现不得通过“是否有行情”反推决策资格。

本轮 `UniverseSnapshot` 保存实际展示采集成员及其来源。`MarketInputSnapshot` 保存所有采集标的的报价、分钟强弱引用和逐数据集质量。历史 K 线和资金流引用仍只来自已经固化的 close/backfill 输入，不在盘中补抓。

## 标的范围与决策隔离

### 展示采集范围

以下标的进入 `capture_symbols`：

- 手动持仓台账中的所有持仓。
- 当日活动计划 `symbol_contexts` 中的所有标的。
- 当前观察池中 `plan_enabled=true`、证券类型为 A 股或 ETF、结算制度已验证且证券仍有效的标的。

同一标的只采集一次，并保留全部来源。adapter 返回的额外证券继续忽略并告警。

### 决策范围

`decision_symbols` 只包含当前持仓和有效活动计划标的。账户估值只读取持仓；策略、风控、建议和通知只遍历 `decision_symbols`。

计划外非持仓观察标的即使报价完整、分时强弱为 strong，也不能生成 `buy/add/sell/reduce/hold/watch/avoid` 建议。行情页可以显示其数据质量和强弱，但不得将展示状态描述为交易动作。

## 运行时钟与幂等周期

当前调度器把冻结的 `task_now` 同时用作调度周期和运行时钟，导致开始与结束时间相同。修复后分离：

- `as_of`：调度器触发时的逻辑时间，用于交易日、交易分钟和三分钟周期幂等键。
- `clock`：每次调用都返回真实当前时间，用于 `started_at`、`finished_at`、`fetched_at` 和审计时间。
- provider 耗时继续使用单调时钟 `perf_counter`。

`run_intraday(as_of=task_now)` 根据 `as_of` 形成稳定的 `intraday:trade_date:HHMM` 幂等键；工作流内部使用真实 `clock` 记录执行时间。测试可以分别注入确定的逻辑时间和可推进时钟，不依赖休眠。

完成状态必须满足 `finished_at >= started_at`。执行了实际 provider 调用的运行应能观测到正耗时；极短的无调用路径允许小于 1ms，但不得再由冻结时间强制为零。监控继续从开始和结束时间计算 `duration_ms`。

## 分钟数据源降级

`AkShareIntradayProvider` 和 ETF 对应 provider 保持同一内部 Protocol，并在 adapter 内执行以下顺序：

1. 首选现有 AkShare 东方财富 1 分钟接口。
2. 首选接口发生请求异常时，调用 AkShare 新浪 `stock_zh_a_minute`。
3. A 股和 ETF 都使用后端元数据确定的交易所前缀调用新浪接口，不按名称猜测。
4. 将新浪的 `day/open/high/low/close/volume/amount` 映射为内部 `MinuteBar`。
5. 新浪 `volume` 已是股，不再乘以 100；金额保持人民币元。
6. 继续执行六位代码、有限数值、非负成交量、交易日、交易分钟和时区校验。
7. fallback 成功时每条分钟线保存明确 source，例如 `akshare_sina_minute`。
8. 两个接口都失败时抛出清理后的 `MarketProviderError`，不得泄露底层 URL、凭据或原始响应。

fallback 是 provider 能力降级，不形成策略或服务层直连第三方。单个标的双失败只写该标的 `minute_bar` 和 `intraday_strength` failed 结果，并继续处理其他标的。

## 数据质量与运行状态

逐标的质量继续按数据集保存。展示标的与决策标的使用同一质量模型，不伪造成功：

- 报价或分钟数据完整时保存 complete。
- 可用但字段不完整时保存 degraded。
- 超过配置的有效交易分钟阈值时保存 stale。
- 两个分钟来源均失败时保存 failed。

任一展示标的数据失败可以使整轮为 degraded，但不得阻断其他标的，也不得撤销已完成的持仓风险建议。所有决策标的关键报价均不可用时仍按现有规则失败或降级。

若 `capture_symbols` 为空，不调用 provider，并保存“没有符合条件的盘中采集标的”警告。调度状态使用明确的 no-eligible-symbols 原因，不能把它描述为完成了行情刷新。

运行监控中的：

- `requested_symbols` 记录 `capture_symbols` 数量。
- `processed_symbols` 记录实际形成持久化质量结果的标的数。
- `provider_calls/provider_duration_ms/rows_received/rows_written` 记录真实采集成本。
- `recommendation_count` 只可能来自 `decision_symbols`。

## API 与前端行为

现有行情读取 API 路径保持不变。行情页的自动刷新继续只读取数据库，不触发 provider。

观察标的完成盘中采集后：

- 扫描器显示当前报价、涨跌幅、分时强弱、数据时间和质量。
- 概览、分钟线、分时强弱和数据引用读取本轮快照。
- 没有活动计划时，计划状态保持为空，不显示为采集错误。
- 没有建议时保持为空，不从强弱标签推导建议。

日 K 或资金流尚未首次回填时，后端继续诚实返回 unavailable 和空集合，并给出稳定 warning。前端把这类响应渲染为“尚未完成首次回填”的中性空状态；真正的 provider failed/stale 仍使用警告或错误状态。ETF 资金流继续显示 not_applicable，不视为待回填或失败。

## 测试设计

### 工作流测试

- 无持仓、无活动计划、存在已启用观察标的时，观察标的进入 `capture_symbols` 并调用报价和分钟 provider。
- 同一场景不调用策略、风控、建议持久化或通知分发，并且 `recommendation_count=0`。
- 有持仓或活动计划时，决策标的继续完成现有账户、策略、风控和建议链。
- 展示标的失败时，其他标的和持仓风险管理继续执行，运行状态与逐数据集计数正确。
- 空展示采集范围不调用 provider，并记录明确的调度原因。

### 门禁测试

- 无活动计划的非持仓观察标的即使具有完整报价、strong 分时和其他强因子，也不能生成 `buy/add`。
- 活动计划存在时，原有计划条件、多因子和硬性风控测试继续通过。
- 手动持仓和手动资金账户的最新权威值继续优先。

### Adapter 测试

- 东方财富分钟请求成功时仍使用首选数据。
- 首选请求失败、Sina 成功时，正确映射 A 股和 ETF 字段、交易所前缀、时区、交易分钟、股数单位和 source。
- 两个来源均失败时抛出稳定、脱敏的 provider 错误。
- 非法数值、错误交易日和非交易分钟不能进入内部模型。

### 时钟与监控测试

- 调度 `as_of` 决定稳定的三分钟 run ID 和幂等键。
- 可推进真实时钟产生 `finished_at > started_at` 和正 `duration_ms`。
- provider 计数、provider 耗时、请求/处理标的数和行数按展示采集范围统计。
- 重复触发同一周期仍复用既有运行，不重复调用 provider。

### API 与前端测试

- 行情 symbols、overview、minute-bars、intraday-strength 和 trace 返回最新展示采集事实。
- 日 K 和资金流未回填时显示中性待回填状态。
- stale、failed、ETF not_applicable 三种状态保持可区分。
- 桌面扫描器和移动标的抽屉均能查看观察标的数据，不出现布局重叠。

## 验收标准

- 在无持仓、无活动计划但存在已启用观察标的的盘中轮次中，最新 capture run 的 `requested_symbols` 大于零、`provider_calls` 大于零，并形成当日报价和分钟质量记录。
- 行情 API 能读取这些观察标的的当日报价、分钟线和分时强弱。
- 同一轮不产生任何计划外观察标的建议或通知。
- 监控中的运行开始、结束和耗时来自真实时钟，不再固定为 `0ms`。
- 东方财富分钟接口失败而 Sina 可用时，分钟采集仍成功且 source 可追溯。
- Python 回归测试、前端单元测试、前端构建和目标 Playwright 场景通过。
- 文档同步反映展示采集范围与决策范围的区别。
