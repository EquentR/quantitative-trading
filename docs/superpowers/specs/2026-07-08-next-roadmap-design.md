# 后续五阶段路线 Spec

## 背景

本项目是个人 A 股短线量化决策辅助系统。系统只输出可解释建议，不自动真实下单，不控制真实交易客户端，不读取真实券商凭据，不承诺确定性收益。

当前仓库已经具备这些基础能力：

- 手动持仓台账。
- 手动资金账户。
- 账户估值快照。
- AkShare 行情报价 adapter，并正在补充东方财富公开单票行情和腾讯 quote 兜底能力。
- SQLite 存储。
- CLI。
- FastAPI HTTP API。
- 后台账户快照调度。
- Vue 本地控制台，覆盖认证、仪表盘、准备、监控、复盘占位和设置。

下一阶段目标不是继续堆 UI，而是围绕首版核心目标推进：稳定生成可追溯建议，并让 CLI、后台、HTTP API 和 Web 继续共享同一套核心逻辑。

## 规约边界

本 spec 必须服从：

- `docs/project-spec.md`
- `docs/trading-policy.md`
- `docs/data-sources.md`
- `docs/recommendation-contract.md`

所有阶段都必须遵守以下边界：

- 不实现自动真实下单。
- 不模拟点击或控制真实交易客户端。
- 不读取、保存、提交真实账户密码、token、cookie、API key。
- 东方财富/妙想等第三方数据源 API key 只能作为本地数据源凭据管理，不得作为真实券商账户凭据使用；不得写入 git、日志、错误响应或前端持久化存储。
- 不从东方财富模拟组合推断真实持仓、成本、数量、可用数量、现金余额或净本金。
- 手动持仓台账是真实持仓权威源。
- 手动资金账户是现金余额、净本金和可用买入资金权威源。
- 外部数据源必须通过 adapter 接入，策略、风控和建议模块不得直接依赖第三方接口字段。
- 风控优先于策略信号。
- 每条建议必须包含理由、风险、失效条件、仓位约束和数据时间戳。
- Web 只消费后端 API 和维护本地手动数据，不在前端推导买卖建议。

## 总体交付顺序

五个阶段按依赖顺序推进：

1. P0：当前收口和工具链稳定。
2. P1：股票池、数据 adapter 和缓存。
3. P2：收盘计划、规则策略、风控和建议契约。
4. P3：CLI/API/调度/审计/本地通知闭环。
5. P4：Web 接入计划、建议、审计和人工反馈。

每个阶段单独审核、单独实现、单独验证。未完成前一阶段验收前，不进入后续阶段的大规模实现。

## P0：当前收口和工具链稳定

### 目标

把当前工作区和测试链路收稳，确保后续开发有可重复验证基础。

### 范围

后端：

- 收尾当前行情 adapter 兜底改动，保持兜底逻辑只在 `market` adapter 边界内返回内部 `QuoteSnapshot`。
- 收尾成本价三位小数相关 API 和测试。
- 确认现有 Python 测试保持通过。

Web：

- 使用 nvm 固定和切换前端 Node 版本。
- 增加 `.nvmrc`，内容固定为 `24`。
- 在 `src/web/package.json` 增加 `engines.node`，要求 `>=24 <25`，和 `.nvmrc` 保持一致。
- 将 `src/web/package.json` 中所有前端直接依赖和开发依赖从 `latest` 收敛为精确版本号。
- 后续依赖升级必须显式修改版本号、更新 `pnpm-lock.yaml`，并重新运行前端测试。
- 确认 `pnpm -C src/web test` 可以在 nvm 指定 Node 版本下运行。

文档：

- README 或前端开发文档说明：
  - `nvm use`
  - `pnpm install`
  - `pnpm -C src/web test`
  - `pnpm -C src/web dev`

### 非目标

- 不新增交易建议能力。
- 不新增 Web 交易决策页面。
- 不重构现有 API 结构。

### 验收标准

- Python 测试通过：`.venv/bin/python -m pytest -q`。
- 前端测试通过：`pnpm -C src/web test`。
- Web 开发者可以通过 `nvm use` 切换到 Node 24。
- `src/web/package.json` 不再包含 `latest` 版本声明。
- 未提交真实密钥、真实账号、真实持仓敏感数据。
- 当前未提交改动按主题整理清楚，避免把工具链修复和业务功能混在一个不可审查变更里。

## P1：股票池、数据 Adapter 和缓存

### 目标

补齐首版股票池和可追溯数据输入，为计划和建议生成提供统一输入。

### 范围

后端模块：

- 新增 `universe` 模块，生成首版股票池。
- 新增或扩展东方财富自选置顶 adapter。
- 新增本地自选置顶维护能力，支持手动新增、编辑、删除、JSON 导入、CSV 导入和 CSV 导出，作为第三方同步失败或不使用第三方同步时的候选池来源。
- 新增东方财富/妙想数据源凭据配置能力，支持环境变量、未入库本地配置和后续 Web 设置入口。
- 新增自选股计划/建议开关维护能力，用于控制哪些自选置顶股票进入后续计划和建议生成。
- 新增数据缓存和快照存储，覆盖自选置顶、股票池、行情快照、历史 K 线、资金流和分时强弱的可追溯记录。
- 继续保持 AkShare、东方财富公开 quote、腾讯 quote 等外部能力只通过 adapter 进入系统。

核心模型：

- `WatchPinnedInput`
  - `symbol`
  - `name`
  - `rank`
  - `plan_enabled`
  - `note`
- `WatchPinnedItem`
  - `symbol`
  - `name`
  - `rank`
  - `plan_enabled`
  - `source`
  - `note`
  - `updated_at`
- `UniverseMember`
  - `symbol`
  - `name`
  - `sources`
  - `priority`
  - `ledger_updated_at`
  - `watch_pinned_rank`
  - `plan_enabled`
  - `plan_enabled_source`
  - `created_at`
- `MarketInputSnapshot`
  - `universe_snapshot_id`
  - `quote_snapshot_refs`
  - `history_snapshot_refs`
  - `money_flow_snapshot_refs`
  - `intraday_strength_snapshot_refs`
  - `data_time`
  - `fetched_at`
  - `warnings`

数据源规则：

- 股票池只包含手动持仓台账股票和东方财富自选置顶股票。
- 持仓来源优先级高于自选置顶来源。
- 东方财富自选置顶可以来自第三方同步结果，也可以来自本地手动维护的自选置顶镜像。
- 手动维护的自选置顶镜像与第三方同步结果进入同一套 `WatchPinnedItem` 契约，并通过 `source` 标记来源。
- `WatchPinnedItem.source` 可取 `manual`、`synced`、`manual_synced`。
- 第三方同步和本地手动维护按 `symbol` 合并。
- 同一股票同时存在于本地和第三方同步结果时，保留本地 `plan_enabled` 和 `note`，用第三方同步结果更新 `name` 和 `rank`，并标记 `source=manual_synced`。
- 第三方新同步进来的股票默认 `plan_enabled=false`，标记 `source=synced`。
- 第三方同步中消失但本地仍手动维护的股票继续保留，标记 `source=manual`。
- 股票池分为“可见股票池”和“决策启用集合”：可见股票池用于查看和维护，决策启用集合用于重行情拉取、收盘计划和后续建议。
- 手动持仓台账股票必须默认进入决策启用集合，不受自选股计划/建议开关影响。
- 自选置顶股票默认只进入可见股票池；只有手动打开计划/建议开关后，才进入决策启用集合。
- 未打开计划/建议开关的自选置顶股票不得拉取历史 K 线、资金流、分时强弱等重数据，也不得生成买入观察计划或建议。
- 自选置顶只用于候选买入观察，不作为真实持仓来源。
- 自选置顶同步失败时，使用上一次成功同步缓存，并标记 stale/warning；如果从未成功同步过，则只使用手动持仓台账生成股票池。
- 任一外部接口失败时，必须记录失败来源、失败摘要和受影响股票，不得静默吞掉。
- 东方财富/妙想 API key 只允许用于数据源 adapter 鉴权和自选置顶同步，不得传入策略、风控、建议、审计 payload。
- 后端只返回凭据配置状态、更新时间和校验结果，不返回 API key 明文。
- Web 设置 API key 时只做一次性提交，不写入 localStorage、Pinia 持久化或任何前端可恢复存储。

CLI/API：

- CLI 支持手动同步自选置顶列表。
- CLI 支持手动新增、更新、删除、导入和导出本地自选置顶镜像。
- CLI 支持查看当前股票池。
- CLI 支持为自选置顶股票打开或关闭计划/建议开关。
- HTTP API 可只读查看股票池和数据源状态。
- HTTP API 支持自选置顶镜像 CRUD、JSON 导入、CSV 导入和 CSV 导出，字段为 `symbol,name,rank,plan_enabled,note`。
- HTTP API 支持维护自选置顶股票的计划/建议开关。
- HTTP API 支持本地设置、重置和连通性测试东方财富/妙想数据源 API key；写入目标必须是 `.gitignore` 覆盖的本地配置或等价的未入库本地密钥存储。

### 非目标

- 不生成交易建议。
- 不做全市场扫描。
- 不接入真实券商账户。
- 不使用东方财富模拟组合推断真实持仓。
- 不在前端展示或保存已配置 API key 明文。

### 验收标准

- 给定手动持仓台账和 fake 自选置顶 adapter，可生成带来源标记的股票池。
- 同一股票同时来自持仓和自选置顶时，结果只保留一条，来源包含两个标记，持仓优先。
- 本地自选置顶镜像支持单条新增、编辑、删除，并支持 JSON/CSV 全量导入导出。
- CSV 字段固定为 `symbol,name,rank,plan_enabled,note`。
- JSON 字段固定为 `symbol`、`name`、`rank`、`plan_enabled`、`note`。
- 第三方同步不得覆盖本地维护的 `plan_enabled` 和 `note`。
- 第三方同步可以更新同一股票的 `name` 和 `rank`。
- 持仓股票默认 `plan_enabled=true`。
- 自选置顶股票默认 `plan_enabled=false`，手动打开后才参与重行情拉取、收盘计划和建议生成。
- 自选置顶同步失败且存在历史缓存时，股票池继续使用历史缓存并显式标记 stale/warning。
- 自选置顶同步失败且无历史缓存时，股票池只包含手动持仓台账股票。
- 缓存记录区分市场数据发生时间和系统获取时间。
- adapter 失败会返回可追溯 warning 或错误状态。
- 策略层不能直接 import AkShare、东方财富 Skill 文件或 requests 调用外部接口。
- 配置 API key 后，状态接口只能显示 `configured`、`missing`、`invalid` 或最近校验摘要，不泄露 key 内容。

## P2：收盘计划、规则策略、风控和建议契约

### 目标

实现“收盘计划 + 盘中触发”的核心交易建议链路，让系统能产出符合契约的可解释建议。

### 范围

后端模块：

- 新增 `planning` 模块，生成次日交易计划。
- 新增 `strategy` 模块，输出规则策略信号。
- 新增 `risk` 模块，执行硬性风控。
- 新增 `recommendation` 模块，合成最终建议。
- 新增内部模型并与 `docs/recommendation-contract.md` 对齐。

收盘计划：

- 输入：
  - 手动持仓台账。
  - 手动资金账户。
  - 账户估值快照。
  - 股票池。
  - 行情、历史 K 线、资金流、分时强弱等缓存。
- 输出：
  - 次日观察列表。
  - 持仓管理计划。
  - 候选买入观察计划。
  - 关键价位。
  - 候选动作。
  - 风险条件。
  - 失效条件。
  - 数据引用。

计划有效期和台账对齐：

- 收盘计划默认适用于下一个 A 股交易日。
- 收盘计划默认有效到适用交易日 `15:00 Asia/Shanghai`，当天未触发的计划条件自动失效。
- 停牌、涨跌停、行情缺失、账户估值不可用、计划依赖数据超过有效时间时，相关建议必须降级或失效。
- 生成次日计划和盘中触发建议前，必须重新读取最新手动持仓台账。
- 次日可用数量、持仓数量、成本价和台账更新时间必须以最新手动持仓台账为准，不得只用前一日收盘计划中的推算值。
- 若最新手动持仓台账与收盘计划引用的台账快照不一致，持仓相关建议必须使用最新台账重新计算；无法重新计算时降级为人工复核或暂停建议。

盘中触发：

- 只检查当前持仓和收盘计划内股票。
- 根据最新行情、关键价位、分时强弱、资金流和计划条件判断是否触发建议。
- 不允许盘中临时扩大到全市场扫描。

规则策略首版方向：

- 以可解释技术规则为主。
- 资金流和分时强弱只作为确认或过滤条件。
- 不把黑盒模型作为核心。
- 每个信号必须有机器可读原因和人类可读原因。
- 首版只覆盖三类信号：
  - 持仓风险管理：止损、止盈或减仓、跌破支撑、T+1 可用数量约束。
  - 持仓继续持有或观察：站上短期均线、未跌破关键位、量能正常。
  - 自选候选买入观察：突破关键位、均线趋势转强、成交额放大，资金流作为确认条件。
- 自选候选买入观察只对 `plan_enabled=true` 的自选置顶股票生效。

风控首版方向：

- 单票仓位上限。
- 总仓位上限。
- 单日新增买入上限。
- 可用买入资金上限。
- 单日交易次数上限。
- T+1 可用数量约束。
- 流动性不足降级或禁止买入。
- 行情缺失或账户估值不完整时降级或暂停依赖完整仓位比例的买入建议。
- 成本价、持仓数量或可用数量缺失时，持仓相关买卖建议降级为人工复核或暂停。

首版风控默认配置：

- 单票仓位上限：总资产的 30%。
- 总仓位上限：总资产的 80%。
- 单日新增买入上限：总资产的 20%。
- 单日交易次数上限：3 次。
- 单只自选候选首次建议仓位：总资产的 10% 到 15%。
- 连续亏损冷却：连续 2 笔亏损后暂停新增买入 1 个交易日。
- 流动性不足：成交额低于配置阈值时禁止买入或降级为观察。
- 以上数值均为可配置默认值，不代表收益假设；实现时必须通过配置模型暴露，并在建议中展示实际生效的仓位约束。

建议动作：

- `buy`
- `sell`
- `add`
- `reduce`
- `hold`
- `watch`
- `avoid`

### 非目标

- 不实现自动下单。
- 不执行模拟组合操作。
- 不输出确定性收益承诺。
- 不实现复杂机器学习模型。
- 不实现复杂板块轮动。
- 不实现全市场排序或全市场机会扫描。

### 验收标准

- 每条最终建议满足 `docs/recommendation-contract.md` 必填字段。
- 每条买入、加仓、持有建议都有失效条件。
- 每条持仓相关建议保留台账更新时间和行情数据时间。
- 买入和加仓建议必须包含资金账户和账户估值上下文。
- 风控拒绝时，策略强信号不能绕过风控。
- 行情或账户估值不可用时，建议必须降级、暂停或显式标注数据缺口。

## P3：CLI/API/调度/审计/本地通知闭环

### 目标

把 P2 的核心能力接入统一运行入口，形成可操作、可追溯、可复盘的首版后台闭环。

### 范围

CLI：

- 生成某交易日收盘计划。
- 查看最新计划。
- 手动触发一次盘中扫描。
- 查看当前建议。
- 查看审计记录。
- 记录人工执行反馈：
  - 是否执行。
  - 实际成交价。
  - 成交数量。
  - 主观备注。
- 人工执行反馈只记录用户实际操作结果和复盘事实，不自动修改手动持仓台账、成本价、持仓数量、可用数量或手动资金账户。
- 需要更新真实持仓或资金口径时，用户必须通过手动持仓台账或手动资金账户维护入口显式修改。

HTTP API：

- 计划只读和生成接口。
- 建议列表和建议详情接口。
- 审计日志查询接口。
- 通知摘要和最近输出查询接口。
- 通知处理状态更新接口，支持标记已读和查询待处理通知。
- 人工执行反馈写入接口。
- 数据源状态和最近缓存状态查询接口。

调度：

- 将当前单一账户快照任务扩展为明确任务类型：
  - 账户快照任务。
  - 收盘计划任务。
  - 盘中触发检查任务。
- 调度状态需要记录任务类型、原因、开始时间、结束时间、状态、错误摘要、关联计划或建议 ID。
- 任务异常不得导致 HTTP 服务退出。

调度默认时间表：

- 账户快照任务：盘中每 3 分钟一次，默认沿用 `QT_INTRADAY_INTERVAL_SECONDS`。
- 收盘计划任务：交易日 `15:30 Asia/Shanghai` 运行一次。
- 盘中触发检查任务：交易日 `09:35-11:30`、`13:00-14:55 Asia/Shanghai`，每 3 分钟一次。
- 启动恢复：如果服务启动时调度状态为 enabled，则恢复账户快照、收盘计划和盘中触发检查任务。
- 如果当天还没有有效收盘计划，盘中触发检查只处理持仓风险，不做自选候选买入观察。
- 节假日和非交易日首版使用可配置交易日历或手动开关兜底；后续可接自动交易日历 adapter。
- 所有调度时间、间隔和交易日历来源必须可配置。

审计：

- 记录输入数据快照引用。
- 记录手动持仓台账快照引用。
- 记录手动资金账户快照引用。
- 记录策略信号。
- 记录风控结果。
- 记录最终建议。
- 记录推送渠道和推送状态。
- 记录人工反馈。

通知：

- 首版实现本地控制台摘要输出、Web 可查看通知摘要和 JSONL 结构化日志。
- 控制台输出面向人工快速阅读，只展示股票、动作、置信度、关键价位、核心理由、主要风险和数据时间。
- Web 通知摘要通过后端 HTTP API 获取，不直接读取本地 JSONL 文件。
- Web 通知摘要至少展示股票、动作、置信度、关键价位、核心理由、主要风险、数据时间和审计引用。
- 通知处理状态由后端本地数据库持久化，首版状态为 `unread`、`read`、`feedback_recorded`。
- 新生成的通知摘要默认 `unread`。
- 用户在 Web 或 CLI 中标记已读后，状态变为 `read`。
- 用户记录人工执行反馈后，关联建议或通知状态变为 `feedback_recorded`。
- 通知处理状态只表示本地处理进度，不代表真实交易完成。
- JSONL 结构化日志保存完整建议、风险、失效条件、仓位约束、输入数据引用、推送状态和人工反馈引用。
- 远程推送接口只预留边界，不在本阶段实现 Telegram、飞书、钉钉或邮件。
- JSONL 日志不得包含密码、token、cookie、API key 或真实券商凭据。

### 非目标

- 不实现远程推送全渠道。
- 不接入 Telegram、飞书、钉钉或邮件凭据。
- 不实现东方财富模拟组合自动操作。
- 不让 API 写操作代表真实交易。
- 不让人工执行反馈自动改写手动持仓台账或手动资金账户。

### 验收标准

- CLI、API、后台调度共享同一套 planning/strategy/risk/recommendation/audit service。
- 同一输入数据通过 CLI 和 API 生成的建议语义一致。
- 每条建议能追溯到输入快照、策略信号和风控结果。
- 控制台通知和 Web 通知摘要包含简洁摘要，JSONL 结构化日志包含完整建议契约和审计引用。
- Web 通过 HTTP API 可以查看最近通知摘要和关联审计记录。
- Web 和 CLI 可以查看 `unread` 待处理数量，并能将通知标记为 `read`。
- 人工反馈写入后，关联通知状态变为 `feedback_recorded`。
- 人工反馈不会被模拟组合操作结果替代。
- 人工反馈写入后不会改变持仓数量、可用数量、成本价、现金余额或净本金。
- 结构化日志不包含密码、token、cookie、API key 或真实券商凭据。

## P4：Web 决策工作台接入

### 目标

在后端计划、建议、审计和反馈 API 落地后，把 Web 从“资源维护控制台”扩展为“本地决策工作台”，但仍不在前端实现交易逻辑。

### 范围

Web 页面：

- 今日仪表盘增加：
  - 最新计划状态。
  - 当前有效建议摘要。
  - 最近通知摘要。
  - 未读和待处理通知数量。
  - 数据质量状态。
  - 待处理人工反馈。
- 准备页增加：
  - 自选置顶同步状态。
  - 股票池查看。
  - 本地自选置顶镜像的新增、编辑、删除、JSON 导入、CSV 导入和 CSV 导出。
  - 自选置顶股票计划/建议开关维护。
  - 数据源状态。
- 监控页增加：
  - 收盘计划任务状态。
  - 盘中触发任务状态。
  - 最近任务错误和数据缺口。
- 复盘页接入：
  - 推荐记录。
  - 人工执行反馈。
  - 审计日志。
  - 通知摘要和推送状态。
  - 输入快照引用。
- 设置页增加：
  - 前端显示偏好。
  - 表格密度。
  - 本地 API 连接配置。
  - 东方财富/妙想数据源 API key 的本地设置、重置和状态检查入口。

Web 交互：

- 建议列表只展示后端返回内容。
- 建议页面首版采用“列表 + 详情抽屉”结构。
- 建议列表用于快速扫视，展示股票、动作、置信度、处理状态、关键价位、数据时间。
- 建议详情抽屉展示完整建议契约，包括理由、风险、失效条件、仓位约束、账户上下文、持仓上下文、数据引用和审计引用。
- 建议详情必须展示理由、风险、失效条件、仓位约束和数据时间。
- 建议和通知列表支持 `unread`、`read`、`feedback_recorded` 状态展示和筛选。
- 用户可以在 Web 中将通知标记为已读；该操作只更新本地处理状态，不修改建议内容、持仓台账或资金账户。
- 人工反馈写操作文案使用“记录人工执行反馈”，不得使用“成交成功”“买入成功”“卖出成功”等真实交易完成语义。
- 人工反馈提交成功后，可以提示用户前往手动持仓台账或资金账户维护入口，但不得自动修改台账或资金账户。
- 对 partial/unavailable 数据必须显式展示警告，不得正常高亮账户级估值。
- 数据源 API key 设置表单必须说明“仅用于本地第三方数据源访问，不代表真实券商登录或交易授权”。
- API key 提交成功后只显示配置状态，不回显明文；退出页面后前端无法恢复该 key。
- 自选股计划/建议开关文案必须说明“打开后会拉取更多行情数据并参与计划和建议生成，关闭后仅保留在可见股票池中”。
- 自选置顶镜像写操作文案使用“保存自选观察项”“删除本地自选记录”“导入自选观察项”，不得暗示真实交易或真实持仓变化。

Web 工程质量：

- 统一 `ApiErrorAlert`。
- 字段级错误展示。
- 成功 toast 文案遵守安全边界。
- 长股票名称、长 warning、长错误信息不撑破布局。
- 桌面和窄屏都要通过视觉或 E2E 冒烟验证。

### 非目标

- 不在 Web 中计算策略信号。
- 不在 Web 中生成买卖建议。
- 不实现真实交易按钮。
- 不保存真实券商凭据。
- 不在前端持久化保存东方财富/妙想 API key。
- 不展示假收益表现。
- 不在首版实现复杂图表或技术指标可视化。

### 验收标准

- Web 展示的计划、建议、审计、反馈均来自后端 API。
- Web 展示的通知摘要和推送状态均来自后端 API。
- token 过期、setup required、unauthorized 都有明确跳转和提示。
- Web 可以展示未读通知数量，并支持按处理状态筛选建议和通知。
- Web 标记已读后刷新通知摘要、仪表盘和待处理数量。
- 建议列表和详情抽屉都只展示后端返回的建议契约数据，不在前端补算策略或风险字段。
- 详情抽屉必须能完整展示一条建议的理由、风险、失效条件、仓位约束、上下文和审计引用。
- 人工反馈提交后刷新建议、审计和仪表盘相关 query。
- 人工反馈提交后，持仓台账和资金账户数据不应自动变化；如需变化，必须由用户通过对应维护表单显式提交。
- 建议详情中缺少风险、失效条件或数据时间时，前端必须以错误或不可用状态展示，不应当作正常建议渲染。
- 数据源 API key 设置、重置和状态检查都有明确成功/失败提示，且测试覆盖不回显、不落 localStorage。
- 自选置顶股票开关变更后刷新股票池、数据源状态、计划相关 query。
- 前端测试和关键 E2E 冒烟测试通过。

## 跨阶段数据流

目标数据流如下：

1. 手动持仓台账和手动资金账户由 CLI/API/Web 维护。
2. 股票池由手动持仓台账和东方财富自选置顶构建。
3. 数据 adapter 拉取行情、历史 K 线、资金流和分时强弱。
4. 缓存层保存输入快照和数据时间。
5. 收盘计划读取股票池和数据缓存，生成次日计划。
6. 盘中触发只读取持仓和计划内股票。
7. 策略模块生成候选信号。
8. 风控模块过滤或降级候选信号。
9. 建议模块生成契约化建议。
10. 通知和审计模块记录输出和人工反馈。
11. CLI/API/Web 都只消费这些共享 service。

## 审核方式

建议按阶段审核：

1. 先审核 P0 是否足够收口，尤其是 nvm、Node 版本、pnpm 依赖锁定和当前未提交改动归类。
2. 再审核 P1 的股票池和数据缓存边界。
3. 再审核 P2 的策略、风控和建议契约。
4. 再审核 P3 的 CLI/API/调度/审计闭环。
5. 最后审核 P4 的 Web 接入范围。

每一阶段审核通过后，再为该阶段单独写实现计划。实现计划应拆成可测试的小任务，不把五个阶段作为一次性大改实现。
